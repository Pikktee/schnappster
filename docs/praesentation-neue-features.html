Daten-Isolation

Doppelte Absicherung

  1. Anwendungsebene: Jede API-Abfrage filtert nach owner_id = aktueller Nutzer.
  2. Datenbankebene: PostgreSQL Row-Level Security (RLS) — selbst bei einem Bug im Code kommt kein Nutzer an fremde Daten.

Rollen-System: User (eigene Daten) und Admin (globale Einstellungen).

Row-Level Security (RLS)

PostgreSQL filtert automatisch — Nutzer sehen nur ihre Zeilen.

Wie funktioniert das?

  1. Nutzer meldet sich an → erhält ein JWT (JSON Web Token) von Supabase.
  2. Bei jeder API-Anfrage wird die User-ID aus dem Token in die Datenbank-Session geschrieben.
  3. PostgreSQL-Policies prüfen automatisch: owner_id = JWT.sub.
  4. Nur passende Zeilen werden zurückgegeben — transparent und ohne extra Code.

Remote MCP-Server

KI-Assistenten bedienen Schnappster über das Internet.

Was ist MCP? — Model Context Protocol: ein Standard, über den KI-Assistenten auf externe Dienste zugreifen.
Authentifizierung — Supabase OAuth 2.1: Der KI-Client (z. B. Cursor) öffnet einen Browser-Login. Kein manuelles Token-Kopieren.
Transport — Streamable HTTP (kein WebSocket). Cloudflare Quick Tunnel für die Entwicklung.

MCP Apps

Visuelle Oberflächen direkt im KI-Chat — ohne Browser-Wechsel.

Konzept

MCP Apps sind HTML-Seiten, die der MCP-Server als Ressourcen bereitstellt. Der KI-Client (z. B. Claude) rendert sie inline — der Nutzer sieht eine interaktive Ansicht statt reinem Text.

Drei eingebettete Views

  1. Schnäppchen-Detail — Anzeige mit Bildern, Score, Beschreibung und KI-Begründung.
  2. Schnäppchen-Liste — Tabellarische Übersicht der besten Funde, sortiert nach Score.
  3. Suchaufträge — Verwaltung aller gespeicherten Suchanfragen.

Technik: Jinja2-Templates → HTML+JS, ausgeliefert als text/html;profile=mcp-app.
