Schnappster

Kleinanzeigen.de Schnäppchen-Finder mit KI

Was ist Schnappster?

Eine persönliche Web-App, die Kleinanzeigen.de-Suchergebnisse periodisch abruft, mit KI auf Schnäppchen prüft und über die Ergebnisse informiert.

Kurz: Automatischer Schnäppchen-Finder mit KI-Bewertung.

Zielsetzung

  • Suchaufträge definieren:
    • URL der Kleinanzeigen-Ergebnisliste
    • Filterungsmöglichkeiten (Min/Max-Preis, Blacklist-Schlüsselwörter)
    • Einstellung des Scraper-Intervals.
  • Periodisches Scraping der Kleinanzeigen-Ergebnislisten im Hintergrund.
  • KI-Bewertung: Gescrapte Anzeigen sollen an ein KI-Textmodell gesendet werden. Das Textmodell liefert die Einschätzung zurück, ob die Anzeige ein Schnäppchen ist (Bargain-Score von 0–10).
  • Web-Frontend mit Dashboard, Schnäppchen-Liste, Verwaltung der Suchaufträge, Einstellungen, Logs.
  • Benachrichtigungen: Gefundene Schnäppchen werden automatisch an einen Telegram-Bot gesendet.

Technischer Überblick

Backend (Python)

  • Basis: Python 3.13, FastAPI, SQLModel, uv
  • Datenbank: SQLite (data/schnappster.db)
  • Scraping: curl-cffi (HTTP), BeautifulSoup (HTML parsen), lxml (Parser)
  • Scheduling: APScheduler (Scrape + Analyse im Hintergrund)
  • KI: OpenAI-kompatible API (z. B. OpenRouter)

Frontend

  • Next.js 16, React 19, Tailwind v4, shadcn/ui

Deployment

Produktion uv run start

Backend-Architektur

Schichten im Python-Projekt (von außen nach innen)

app/core/ — DB, Config (.env), APScheduler, Logging
app/models/ — SQLModel-Tabellen + API-Schemas
app/scraper/ — HTTP/HTML (curl-cffi, BeautifulSoup), keine Logik
app/services/ — ScraperService, AIService, SettingsService
app/routes/ — FastAPI-Router (api_router)
cli/ — Einstiege: start, scrape, analyze, dbreset

Scraping und KI-Analyse laufen in getrennten Single-Worker-Queues (kein Überlappen).

Ablauf: Scraping → KI

Zwei Phasen: ScraperService speichert neue Anzeigen, AIService wertet sie aus.

Phase 1 — Scraping

Für jeden fälligen Suchauftrag (AdSearch):

  1. Suchseiten fetchen → Preview-Liste (external_id, URL) parsen.
  2. Bekannte ausfiltern (external_id schon in DB für diesen AdSearch).
  3. Detailseiten der neuen Previews parallel laden → ScrapedAdDetail.
  4. Filtern: Preis (min/max), Blacklist-Keywords, Verkäufertyp, Mindest-Rating.
  5. Neue Ads in DB speichern; bei neuen Ads wird Phase 2 gequeued.

Phase 2 — KI-Analyse

Eigener Job (Analyzer-Queue, kein Überlappen mit Scrape):

  1. Bis zu 10 unverarbeitete Ads (is_analyzed=false), älteste zuerst.
  2. OpenAI-kompatible API: Bewertung (bargain_score 0–10), Kurztext, Begründung.
  3. Bei verbleibendem Backlog: nächsten Analyze-Job in Queue stellen (wiederholt sich bis Backlog leer).

Frontend in Kürze

Next.js + React, statisch gebaut, eine klare Struktur.

Stack — Next.js 16, React 19, Tailwind v4, shadcn/ui. Build: Static Export (web/out/).
API & Typen — Alle Aufrufe in web/lib/api.ts, Typen in web/lib/types.ts.
Layout — Routen unter web/app/(app)/, gemeinsame Sidebar für die App.
Dynamische Routen — /ads/[id], /searches/[id]: FastAPI liefert eine Shell, der Client lädt die echten Daten per API.
